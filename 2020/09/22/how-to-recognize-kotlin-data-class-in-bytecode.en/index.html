How to Recognize Kotlin Data Classes in Bytecode

Sep 22, 2020 | Computer Science / Kotlin | 1.8k Words | 11 Minutes

A colleague working on bytecode analysis recently asked me: “How can I use ASM to identify Kotlin data classes?” I wondered what use case would require distinguishing data classes from regular classes. Turns out, the goal was to extract all classes implementing Serializable (both Java classes and Kotlin data classes) from a project, in order to visualize POJO caches stored in Redis.

After studying the bytecode of a data class, I found several distinctive characteristics. Take this example:

From the bytecode above, we can identify several characteristics of a data class:

  1. The class is final
  2. The constructor takes parameters, and the number of parameters equals the number of getters
  3. It overrides equals(Object), hashCode(), and toString()

Additionally, there are several special methods:

  1. copy(String, int)
  2. copy$default(User, String, int, int, Object)
  3. component1()
  4. component2()

Are these truly the hallmarks of a Kotlin data class? Let’s check the official Kotlin documentation on data classes:

The compiler automatically derives the following members from all properties declared in the primary constructor:

  1. equals() / hashCode() pair
  2. toString() of the form "User(name=John, age=42)"
  3. componentN() functions corresponding to the properties in their order of declaration
  4. copy() function

So with these characteristics, it’s straightforward to distinguish data classes from regular classes. For bulletproof detection, you can also check for copy$default(...) with the ACC_SYNTHETIC flag.
